# development.md — Syncora Development Guide

## Phase Progression

Phases must be completed in order. Each phase has hard exit criteria — do not advance until they pass.

```
Phase 1: Foundation
    └── Phase 2: Database
            └── Phase 3: Auth & RBAC
                    └── Phase 4: Work Order Core
                            └── Phase 5: Dashboard
                                    ├── Phase 6: Notifications
                                    ├── Phase 7: Real-time
                                    └── Phase 8: Mobile + Mapbox ← MOVED HERE
                                            └── Phase 9: Analytics
                                                    └── Phase 10: Maps refinement
```

Phases 6, 7, and 8 can run in parallel once Phase 5 is complete. Phase 9 requires Phase 7 (live data). Phase 10 is self-contained after Phase 7.

## Development Workflow

### Starting a Feature

1. Create a feature branch from `main`:
   ```bash
   git checkout -b feat/work-order-assignment
   ```
2. Implement backend module first (schema → migration → service → controller → guard → tests)
3. Implement frontend after API contract is confirmed (hook → component → page)
4. Run the Definition of Done checklist before opening a PR

### Backend Development Pattern (NestJS)

```
/src
  /modules
    /work-orders
      work-orders.module.ts
      work-orders.controller.ts   ← route handlers, guards applied here
      work-orders.service.ts      ← business logic, audit logging here
      work-orders.repository.ts   ← Prisma calls only
      dto/
        create-work-order.dto.ts
        update-work-order.dto.ts
      guards/
        work-order-ownership.guard.ts
```

Rules:
- Services call repositories — never raw Prisma in controllers
- DTOs validated with class-validator before service receives data
- Guards on controllers, not duplicated in services
- Audit logging in services, not controllers or repositories

### Frontend Development Pattern (Next.js)

```
/app
  /dashboard
    /work-orders
      page.tsx              ← thin shell, composes components
      loading.tsx           ← Skeleton state
      error.tsx             ← NetworkError empty state

/components
  /work-orders
    JobTable.tsx
    JobTable.stories.tsx    ← Storybook: light, dark, empty, mobile variants
    useWorkOrders.ts        ← TanStack Query hook

/lib
  socket.ts                 ← useSocket() hook
  tokens.ts                 ← semantic color token aliases
  roles.ts                  ← role-filtered nav config
```

Rules:
- Pages are thin — data fetching and logic in hooks
- TanStack Query for all server state, Zustand for UI-only state
- Socket events update TanStack Query cache via `queryClient.setQueryData()` — do not manage server data in Zustand

## Real-time Development

When adding a new real-time event:

1. Define the event name as a constant in `lib/socket-events.ts` (shared between FE and BE)
2. Emit from the NestJS Socket.IO Gateway in the relevant service
3. Listen in `useSocket()` and update the correct TanStack Query cache key
4. Trigger the appropriate toast type from `ToastProvider`
5. If the event appends to AlertsPanel, add it to the alert history list

## Testing Strategy

### Backend
- **Unit tests**: service layer (business logic, guard conditions)
- **Integration tests**: controller + guard + service via NestJS testing module
- **Focus areas**: auth flow, RBAC enforcement, ownership scoping, audit log entries

### Frontend
- **Unit tests**: utility functions, hooks (with TanStack Query test utilities)
- **Component tests**: Storybook interaction tests for critical flows (login, status update, toast)
- **Focus areas**: role-filtered nav rendering, socket event → UI update, mobile swipe actions

### Security testing checklist (per PR touching auth/RBAC):
- [ ] Unauthenticated request returns 401
- [ ] Wrong-role request returns 403
- [ ] Customer cannot access another customer's orders
- [ ] Refresh token reuse returns 401 (rotation invalidation works)
- [ ] Expired access token rejected

## UI Development Standards

### Adding a New Status
1. Add token to `globals.css` (`--color-status-<name>`)
2. Alias in `lib/tokens.ts`
3. Add row to `StatusPill.tsx` variant map with icon + token
4. Never use a raw Tailwind color class for status — token only

### Adding a New Page
1. Create `page.tsx`, `loading.tsx`, `error.tsx` under `/app/dashboard/<page>`
2. Add nav item to `lib/roles.ts` with correct role filter
3. Add breadcrumb segment (auto-generated by `useBreadcrumbs()` from route segments)
4. Add empty state component in `/components/empty-states/` if page has a zero-data scenario

### Adding a New Toast Type
Add to `ToastProvider` config:
```ts
{
  type: 'your_event',
  color: 'blue' | 'green' | 'red',
  icon: LucideIcon,
  autoDismiss: 5000 | 10000,  // 10s for red only
  actionLabel: 'View',         // optional
}
```

## Storybook

All components require stories with these variants at minimum:

- Default (light mode)
- Dark mode
- Empty / zero-data state
- Mobile (375px viewport)
- Loading / skeleton state (if applicable)

Map components in Sprint 1: mark stories `[PLACEHOLDER]`. Remove tag when Mapbox is integrated (Phase 10).

## Deployment Notes

- Docker Compose manages PostgreSQL — always run `docker-compose up -d` before starting servers
- Run `npx prisma migrate deploy` (not `dev`) in production environments
- All secrets via environment variables — validate presence at app bootstrap (`ConfigModule` with validation schema)
- Security headers configured at NestJS middleware level — verify with [securityheaders.com](https://securityheaders.com) before first production release
- OWASP Top 10 review required before production go-live
